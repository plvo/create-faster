The interactive CLI provides a guided experience for creating projects. Run the command without any flags to start:

```bash
# Using your favorite package manager
bun create faster
npm create faster
pnpm create faster
```

## Workflow Overview

The CLI walks through configuration in a logical sequence, starting with naming decisions, then application setup, followed by infrastructure choices, and finally installation preferences.

## Step-by-Step Process

<Steps>

<Step>

### Project Name

First, you'll name your project:

```bash
тЧЗ  Name of your project?
тФВ  my-saas
```

The CLI validates that no file or directory with this name exists in the current directory. For single-app projects, this name becomes the application directory. For multi-app projects, it becomes the monorepo root.

</Step>

<Step>

### Blueprint or Custom

If blueprints are available, the CLI asks how you want to start:

```bash
тЧЗ  How do you want to start?
тФВ  тЧП Use a blueprint (pre-composed starter project)
тФВ  тЧЛ Custom (choose everything yourself)
```

**Blueprint**: Skips app configuration, database, and ORM тАФ the blueprint defines all of those. Some blueprints also predefine deployment (e.g., Lambda blueprints). You proceed to deployment (if not predefined), linter, tooling, git, and package manager selection.

**Custom**: Continues with the full configuration flow below.

See [Blueprints](/docs/blueprints) for available starter projects.

</Step>

<Step>

### Application Count

Next, specify how many applications to create:

```bash
тЧЗ  How many apps do you want to create?
тФВ  Eg: a backend + a frontend = enter 2
тФВ  Only a Next.js app = enter 1
тФВ  Turborepo will be used if more than one
тФВ  3
```

**Single app (1)**: Generates a standard framework project structure.

**Multiple apps (2+)**: Automatically configures Turborepo with apps in `apps/` directory and shared packages in `packages/`.

</Step>

<Step>

### App Configuration

For each application, you'll configure its name, stack, and modules.

#### App Name

For single-app projects, the app name matches the project name automatically. For multi-app projects, you'll name each application:

```bash
тЧЗ  Name of the app #1?
тФВ  web

тЧЗ  Name of the app #2?
тФВ  mobile

тЧЗ  Name of the app #3?
тФВ  api
```

#### Stack Selection

Choose the framework for each app. Stacks are organized by type:

```bash
тЧЗ  Select the stack for web
тФВ
тФЬ Web / Mobile App
тФВ  тЧП Next.js (React framework with SSR)
тФВ  тЧЛ TanStack Start (Full-stack React framework)
тФВ  тЧЛ Expo (React Native framework)
тФВ
тФФ Server / API
   тЧЛ Hono (Fast web framework)
   тЧЛ Node.js (Plain TypeScript server)
```

The stack determines the base template and available modules.

#### Module Selection

After selecting a stack, choose optional modules to include. Available modules depend on the selected framework and are grouped by category:

```bash
тЧЗ  Do you want to add any Next.js modules to web?
тФВ
тФВ  UI
тФВ  тЧ╗ shadcn/ui
тФВ  тЧ╗ Next Themes
тФВ
тФВ  Content
тФВ  тЧ╗ MDX
тФВ  тЧ╗ PWA
тФВ
тФВ  Auth
тФВ  тЧ╗ Better Auth
тФВ
тФВ  API
тФВ  тЧ╗ tRPC
тФВ
тФВ  Data Fetching
тФВ  тЧ╗ TanStack Query
тФВ  тЧ╗ TanStack Devtools
тФВ
тФВ  Forms
тФВ  тЧ╗ React Hook Form
тФВ  тЧ╗ TanStack Form
```

Use arrow keys to navigate and space to toggle selections.

</Step>

<Step>

### Database Selection

Choose a database provider or skip:

```bash
тЧЗ  Include a database?
тФВ  тЧП PostgreSQL
тФВ  тЧЛ MySQL
тФВ  тЧЛ None
```

Selecting a database enables ORM selection in the next step. Database configuration is shared across all applications in a monorepo via the `packages/db` package.

</Step>

<Step>

### ORM Selection

If you selected a database, choose an ORM:

```bash
тЧЗ  Configure an ORM?
тФВ  тЧП Drizzle
тФВ  тЧЛ Prisma
тФВ  тЧЛ None
```

The ORM package includes:
- Type-safe database client
- Schema definitions
- Authentication integration (if auth module selected)
- Database connection configuration

</Step>

<Step>

### Deployment Platform

Choose an optional deployment platform:

```bash
тЧЗ  Deployment platform?
тФВ  тЧП SST
тФВ  тЧЛ Terraform (AWS)
тФВ  тЧЛ None
```

**SST**: Generates `sst.config.ts` with stack-aware resource mapping. See [SST](/docs/deployment/sst).

**Terraform (AWS)**: Generates a complete `infra/` directory with AWS provider, S3 backend, and per-environment configs. See [Terraform (AWS)](/docs/deployment/terraform-aws).

</Step>

<Step>

### Linter Selection

Choose a linter/formatter:

```bash
тЧЗ  Code quality tools?
тФВ  тЧП Biome
тФВ  тЧЛ ESLint + Prettier
тФВ  тЧЛ ESLint
тФВ  тЧЛ Prettier
тФВ  тЧЛ None
```

See [Biome](/docs/linter/biomejs) and [ESLint](/docs/linter/eslint) for details.

</Step>

<Step>

### Git Initialization

Choose whether to initialize a Git repository:

```bash
тЧЗ  Initialize Git?
тФВ  Yes / No (default: Yes)
```

If enabled, the CLI runs `git init` after file generation. This is required if you plan to use Husky for git hooks.

</Step>

<Step>

### Tooling

Select additional development tools:

```bash
тЧЗ  Add any extras?
тФВ  тЧ╗ Husky
```

**Husky**: Configures pre-commit hooks with lint-staged (requires Git and a linter). See [Husky](/docs/extras/husky).

</Step>

<Step>

### Dependency Installation

Finally, choose whether to install dependencies immediately and which package manager to use:

```bash
тЧЗ  Install dependencies now?
тФВ  тЧП Install with bun
тФВ  тЧЛ Install with pnpm
тФВ  тЧЛ Install with npm
тФВ  тЧЛ Skip installation
```

If you select a package manager, the CLI runs the installation process:

```bash
тЧЖ  Installing dependencies with bun...
тФВ
тЧЗ  Dependencies installed successfully!
```

If you skip, you can manually run the installation later:

```bash
cd my-saas
bun install
```

</Step>

</Steps>

## Validation Rules

The CLI enforces compatibility requirements during configuration:

**ORM requires Database**: You cannot select an ORM without first selecting a database provider.

**Husky requires Git**: The Husky extra is only available if Git initialization is enabled.

**Unique app names**: In multi-app projects, each application must have a unique name.

If you make incompatible selections, the CLI displays an error message explaining the requirement.
