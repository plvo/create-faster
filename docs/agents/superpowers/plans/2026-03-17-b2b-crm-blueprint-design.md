# Blueprint: b2b-crm

Generic B2B CRM/ERP base with authentication, RBAC, admin panel, and example CRUD.

## Composition

**Apps:**
- `web` (nextjs): shadcn, better-auth, trpc, tanstack-query, tanstack-devtools, tanstack-form, next-themes
- `batch` (node): simple handler template

**Project:** postgres + drizzle

**Repo:** turborepo (2 apps)

## Architecture

### Structure (Flat — Approach A)

No DDD, no domain pattern. Simple flat structure.

```
src/
├── app/
│   ├── (auth)/login/
│   ├── (dashboard)/
│   │   ├── admin/users/
│   │   ├── contacts/
│   │   ├── profile/
│   │   └── page.tsx
├── components/
│   ├── ui/sidebar.tsx          # shadcn primitive
│   ├── navigation/             # app-sidebar, sidebar-links, nav-user, app-header
│   ├── contacts/               # contact-table, contact-form, contact-dialog
│   └── admin/                  # user-table, create-user-dialog, user-actions
├── lib/
│   ├── auth/auth.ts            # better-auth + admin plugin
│   └── constants.ts            # nav routes per role
└── trpc/
    ├── routers/                # user, contact
    └── middleware/rbac.ts      # adminProcedure, managerProcedure
```

### Auth & RBAC

- better-auth admin plugin with two roles: `admin`, `manager`
- `admin` plugin provides: role field, banned/banReason, REST endpoints for user management
- `defaultRole: 'manager'`

**tRPC RBAC middleware:**
- `adminProcedure`: requires `role === 'admin'`
- `managerProcedure`: alias for `protectedProcedure` (admin + manager both access)

**Route protection:**
- `(dashboard)/layout.tsx`: server-side session check, redirect to `/login` if unauthenticated
- Sidebar navigation filtered by role (ADMIN_ROUTES vs MANAGER_ROUTES)

### Pages — Server Components First

All `page.tsx` files are Server Components. Client interactivity via `*.client.tsx` wrapper components.

Pattern:
1. `page.tsx` (server) → prefetch tRPC queries
2. `HydrateClient` → `<Feature>.client.tsx` (client) with Suspense

```
(dashboard)/
├── page.tsx                    # Server → DashboardClient
├── dashboard.client.tsx        # Client: stats cards
├── profile/
│   ├── page.tsx                # Server → ProfileClient
│   └── profile.client.tsx      # Client: edit forms + theme
├── admin/users/
│   ├── page.tsx                # Server → UsersClient
│   └── users.client.tsx        # Client: table + admin actions
└── contacts/
    ├── page.tsx                # Server → ContactsClient
    ├── contacts.client.tsx     # Client: table + create dialog
    └── [id]/
        ├── page.tsx            # Server → ContactClient
        └── contact.client.tsx  # Client: edit form
```

**Pages:**

| Page | Route | Description |
|------|-------|-------------|
| Login | `/login` | Email + password form, redirect to `/` on success |
| Dashboard | `/` | Stats cards (contact count, active users) |
| Profile | `/profile` | Edit account info + change password + theme switcher |
| Admin Users | `/admin/users` | User list + create/ban/unban/reset password (admin only) |
| Contacts | `/contacts` | Contact list + create dialog |
| Contact Detail | `/contacts/[id]` | Contact detail + edit form |

### Sidebar — shadcn Pattern

- `components/ui/sidebar.tsx`: shadcn sidebar primitive (SidebarProvider, Sidebar, SidebarMenu, etc.)
- `components/navigation/app-sidebar.tsx`: composes Header + SidebarLinks + NavUser
- `components/navigation/sidebar-links.tsx`: role-filtered nav items from constants
- `components/navigation/nav-user.tsx`: avatar dropdown (profile, theme toggle, logout)
- `components/navigation/app-header.tsx`: SidebarTrigger + breadcrumbs

### Database Schema

Blueprint adds a `contacts` table alongside the auth tables generated by drizzle + better-auth:

```typescript
export const contactTable = pgTable('contacts', {
  id: text('id').primaryKey().$defaultFn(() => crypto.randomUUID()),
  firstName: text('first_name').notNull(),
  lastName: text('last_name').notNull(),
  email: text('email'),
  phone: text('phone'),
  company: text('company'),
  notes: text('notes'),
  createdById: text('created_by_id').references(() => userTable.id, { onDelete: 'set null' }),
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
});
```

### tRPC Routers

**`contact.ts`** (managerProcedure — admin + manager):
- `list`: query all contacts
- `getById`: query by ID
- `create`: insert with createdById from session
- `update`: update by ID
- `delete`: delete by ID

**`user.ts`** (mixed procedures):
- `me`: protectedProcedure — current user (profile, sidebar)
- `list`: adminProcedure — list all users

Admin mutations (create, ban, reset password) go through better-auth client-side (`authClient.admin.*`), not tRPC.

### Batch App

Override node stack `index.ts` with a batch handler template:
- `BatchEvent` interface (type + payload)
- Switch-based handler pattern
- Local dev execution via `import.meta.url` check

## Decisions

- **No DDD/domain pattern** — flat structure, generic base
- **No recharts** — unlike dashboard blueprint, keep it simple
- **No vitest** — testing setup is a cross-cutting concern for a future `vitest` module, not blueprint-specific
- **Admin mutations via better-auth client** — not duplicated in tRPC
- **Server Components first** — all page.tsx are server components, client interactivity via *.client.tsx
- **sonner** — only extra dependency (toast notifications)
