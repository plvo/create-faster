# Agent Context Generation — Design

**Date**: 2026-05-29
**Status**: Approved (pending implementation plan)

## Problem

Projects generated by create-faster ship with no context for AI dev agents
(Claude Code, Cursor, Codex, Gemini, Copilot). An agent dropped into a fresh
project has to rediscover the stack, conventions, env vars, and how libraries
are wired. This is worse for blueprints, where real application code and
cross-library wiring exist but are undocumented.

Goal: generate AI-agent context in every created project — an `AGENTS.md`
oriented at the project, with per-library detail, and richer pattern guides for
blueprints — without duplicating official library docs.

## Principles

- **Single source of truth per file.** `AGENTS.md` holds the content;
  `CLAUDE.md` is a one-line `@AGENTS.md` import. No filesystem symlink (fragile
  on Windows/WSL). Each tool reads its native file.
- **No AI slop.** Addon fiches describe what is create-faster-specific (local
  config, env, scripts, conventions), never re-explain the library itself.
- **Custom vs blueprint differ in nature, not size.** Custom projects get short
  inline fiches in `AGENTS.md`. Blueprints additionally get pattern guides in
  `docs/agents/` ("how to add a protected route", data flow, wiring).
- **Co-location.** Content lives next to the template it describes, so adding or
  removing a library naturally carries its agent doc.
- **Graceful degradation.** A library without an `__agent.md.hbs` simply
  produces no section. No error, no empty section.

## Output Layout

Single repo:

```
my-project/
├── AGENTS.md              # source of truth (generated programmatically)
├── CLAUDE.md              # contains exactly: @AGENTS.md
└── docs/agents/*.md       # blueprints only (pattern guides)
```

Monorepo (turborepo):

```
my-project/
├── AGENTS.md              # project view: repo, pm, global env, apps + stacks
├── CLAUDE.md              # @AGENTS.md
├── apps/{app}/AGENTS.md   # that app's libraries + conventions
├── apps/{app}/CLAUDE.md   # @AGENTS.md (companion, written next to every AGENTS.md)
└── docs/agents/*.md       # blueprints only
```

A `CLAUDE.md` (`@AGENTS.md`) is written alongside **every** generated
`AGENTS.md`, including per-app ones, so any tool resolving the nearest CLAUDE.md
finds the app context.

## Content Sources

Three content layers, each with a distinct source:

| Block | Source | Applies |
|---|---|---|
| Project header (name, repo, pm, scripts, structure, consolidated env, conventions) | Root template `templates/repo/{type}/__agent.md.hbs`, rendered with full context | Always |
| Addon fiches (local config, create-faster conventions) | `__agent.md.hbs` co-located in the addon's template dir | If addon selected |
| Pattern guides (data flow, "how to add X") | `docs/agents/*.md.hbs` in the blueprint's template dir | If blueprint |

### Co-location convention

```
templates/repo/single/__agent.md.hbs                       # project header (single)
templates/repo/turborepo/__agent.md.hbs                    # project header (mono root)
templates/libraries/better-auth/__agent.md.hbs             # addon fiche
templates/project/orm/drizzle/__agent.md.hbs               # addon fiche
templates/stack/nextjs/__agent.md.hbs                      # stack fiche
templates/blueprints/org-dashboard/docs/agents/auth.md.hbs # rich blueprint guide
```

`__agent.md.hbs` files (header + fiches) are **not** copied as normal output
files. They are collected by the generator, rendered, and assembled into
`AGENTS.md`. Blueprint `docs/agents/*.md.hbs` files **are** normal output files
and flow through the existing Handlebars pipeline.

### Fiche frontmatter (optional)

```yaml
---
scope: app | root        # where the section lands; default: app for libraries/stack, root for project addons
heading: Better Auth      # section title; default: addon's META label
order: 50                 # sort order within AGENTS.md; default by category
---
Config in `src/lib/auth/`. Sessions persisted via {{orm}}.
Add a provider: edit `auth.ts` → `socialProviders`.
```

The body is Handlebars and receives the same context as other templates
(`projectName`, `appName`, `stackName`, `orm`, `database`, `pm`, helpers like
`appPort`, `has`, `hasLibrary`).

## Generation Pipeline

New module `apps/cli/src/lib/agent-context-generator.ts`, modeled on
`env-generator.ts`. Wired into `file-generator.ts` as **step 5** (after
Handlebars templates, after result compilation):

```
generateProjectFiles()
  1. package.json    (programmatic)
  2. .env.example    (programmatic)
  3. Handlebars templates           ← blueprint docs/agents/*.md.hbs render here
  4. compile results
  5. agent context   (programmatic)  ← NEW
```

### Public API

```ts
collectAgentContextFiles(ctx: TemplateContext): { destination: string; content: string }[]
```

Returns the `AGENTS.md` files and their `CLAUDE.md` companions. The blueprint
`docs/agents/*.md` files are NOT produced here — they are normal templates
(step 3).

### Algorithm

1. **Header.** Render `templates/repo/{ctx.repo}/__agent.md.hbs` with full
   context → the top of root `AGENTS.md`. The header template uses existing
   helpers and the `envGroups`/scripts already available in the enriched
   context.
2. **Collect fiches.** For each app: for each selected library compatible with
   its stack, locate `__agent.md.hbs` in that library's template dir; render it
   with the app's context. Do the same for the app's stack, and for project
   addons (database, orm, linter, tooling). Missing file → skipped silently.
3. **Group by scope.** Reuse the scope→path mapping idea from `env-generator`:
   - `root` → root `AGENTS.md`
   - `app` → `apps/{appName}/AGENTS.md` (turborepo) or root `AGENTS.md` (single)
   - Single repo: all scopes collapse to root, sections deduped by heading.
4. **Assemble.** For each destination `AGENTS.md`: header (root only) +
   addon sections sorted by (category, `order`, heading). Per-app files get a
   short generated preamble (app name, stack) + that app's sections.
5. **Blueprint architecture + index.** If `ctx.blueprint`, inject an
   `## Architecture` section (from `META.blueprints[name].agentArchitecture`)
   and a link index to the `docs/agents/*.md` files the blueprint ships.
6. **CLAUDE.md companions.** For every `AGENTS.md` destination, emit a sibling
   `CLAUDE.md` whose content is exactly `@AGENTS.md\n`.

## AGENTS.md Anatomy (custom, single repo)

```markdown
# my-project

Generated with create-faster. Stack: Next.js + Drizzle + PostgreSQL.

## Stack
- App: Next.js (App Router)
- ORM: Drizzle · Database: PostgreSQL
- Linter: Biome · Package manager: bun

## Scripts
- `bun dev` — dev server (port 3000)
- `bun db:push` — push Drizzle schema
- `bun lint` — Biome

## Environment
Variables in `.env` (see `.env.example`): DATABASE_URL, BETTER_AUTH_SECRET…

## Conventions
- Imports: alias `@/*` → `src/*`

## better-auth          <- from __agent.md.hbs
Config in `src/lib/auth/`. Sessions persisted via Drizzle…

## shadcn               <- from __agent.md.hbs
Components in `src/components/ui/`. Add via `bunx shuip add X`…
```

Header (Stack/Scripts/Env/Conventions) comes from
`templates/repo/single/__agent.md.hbs`; the bottom sections come from
co-located addon fiches.

## Blueprint AGENTS.md + docs/agents/

Root `AGENTS.md` gains an Architecture section and a guide index:

```markdown
## Architecture
org-dashboard: multi-tenant SaaS with orgs, roles, dashboard.
Per-aspect detail in `docs/agents/`:
- [Auth & permissions](docs/agents/auth.md)
- [Data layer](docs/agents/data-layer.md)
```

`docs/agents/*.md.hbs` are rich, editable markdown rendered like any blueprint
template (override semantics included).

## META Changes

- Blueprints gain an optional field:

```ts
blueprints: {
  'org-dashboard': {
    // ...existing
    agentArchitecture: 'org-dashboard: multi-tenant SaaS…', // Architecture section body
  }
}
```

- No new field for libraries/stacks/project addons: the presence of a
  co-located `__agent.md.hbs` is the declaration. This keeps `__meta__.ts` lean.

## Types

- `TemplateContext` gains `agentContext: boolean` (default `true`).
- `types/meta.ts`: `MetaBlueprint` gains optional `agentArchitecture?: string`.

## CLI & Activation

- **On by default.** Opt-out flag `--no-agent-context`.
- `flags.ts`: register the commander option, default `true`; thread into the
  partial context.
- No dedicated interactive prompt (avoids friction). Activated silently;
  mentioned in the final summary ("Generated AGENTS.md for AI agents").
- `buildCliCommand()` in `tui/summary.ts`: append `--no-agent-context` to the
  recreate command only when disabled.

## Initial Rollout Scope

Write `__agent.md.hbs` for the high-value subset first; complete over time:

- Stacks: `nextjs`, `expo`, `hono`
- Project addons: `drizzle`, `prisma` (ORM), `postgres`/`mysql` (database)
- Libraries: `better-auth`, `shadcn`, `trpc`
- Root headers: `templates/repo/single/__agent.md.hbs`,
  `templates/repo/turborepo/__agent.md.hbs`

Libraries without a fiche degrade cleanly (no section). Trivial libs
(`next-themes`, `pwa`, etc.) can be added later or left out.

  - single repo: collapse + dedup by heading
  - turborepo: per-app destinations

  - CLAUDE.md companion content is exactly `@AGENTS.md`
- Integration: generate one custom project and one blueprint; assert

  present only for the blueprint.

## Out of Scope (YAGNI)

- `.claude/skills/` generation (Claude-only; revisit if auto-trigger proves
  valuable on top of AGENTS.md).
- Cursor/Gemini-specific files beyond AGENTS.md (the standard already covers
  them).
- Pulling content from the planned MDX docs site (no dependency on
  not-yet-built infrastructure).
